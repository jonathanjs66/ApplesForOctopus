# מצגת תרגול בעברית: Octopus Fruits App

## 1. פתיחה קצרה

שלום, בניתי אפליקציית ווב פשוטה שמציגה את כמות התפוחים מתוך מסד נתונים MongoDB.

המטרה שלי בפרויקט הייתה לא רק לגרום לאפליקציה לעבוד, אלא גם להציג חשיבה מסודרת של DevOps:

- הפרדה בין רכיבים
- פריסה מלאה באמצעות קוד
- בדיקות אוטומטיות
- תהליך CI/CD
- הפרדת הרשאות למסד הנתונים
- יכולת גיבוי ושחזור

---

## 2. מה האפליקציה עושה

האפליקציה מציגה עמוד HTML פשוט עם הודעת hello world ועם הנתון:

"We have 5 apples in stock."

המספר לא כתוב ידנית בעמוד, אלא נשלף מתוך MongoDB.

כלומר, ה-Node.js app קורא מהקולקציה `fruits`, מוצא את הרשומה של `apples`, ומציג את הערך של `qty`.

---

## 3. הארכיטקטורה שבניתי

המערכת בנויה מ-3 קונטיינרים נפרדים:

- `nginx`
- `app`
- `mongodb`

זרימת הבקשה היא:

Browser -> Nginx -> Node.js App -> MongoDB

### למה בחרתי בהפרדה הזאת

הפרדתי בין הרכיבים כדי שכל רכיב יעשה תפקיד אחד ברור:

- `nginx` הוא שער הכניסה
- `app` מכיל את הלוגיקה האפליקטיבית
- `mongodb` שומר את הנתונים

זה גם תואם לדרישת המשימה שכל רכיב ירוץ בקונטיינר נפרד.

---

## 4. למה Nginx מקדימה את Node.js

בחרתי לשים Nginx מול האפליקציה כדי לבנות מבנה שיותר דומה לסביבת production.

היתרונות:

- נקודת כניסה אחת למערכת
- אפשרות להרחיב בעתיד ל-load balancing
- הפרדה בין reverse proxy לבין הלוגיקה של האפליקציה

גם אם אפליקציית Node קטנה ופשוטה, המבנה הזה יותר מקצועי מאשר לחשוף ישירות את Node החוצה.

---

## 5. איך הנתונים נטענים ל-MongoDB

בזמן עליית MongoDB, אני מריץ סקריפט אתחול שמכניס את הנתונים שניתנו במשימה:

- apples
- bananas
- oranges
- avocados

בנוסף לכך, סקריפט האתחול גם יוצר משתמש ייעודי לאפליקציה עם הרשאות `readWrite` רק על `fruitsdb`.

### למה זה חשוב

כי לא רציתי שהאפליקציה תתחבר למסד הנתונים עם משתמש אדמין.

זו הפרדה נכונה יותר של הרשאות:

- משתמש אדמין לצורכי ניהול
- משתמש אפליקטיבי לצורכי runtime בלבד

---

## 6. איך האפליקציה מתחברת למסד הנתונים

האפליקציה בנויה ב-Node.js עם Express ו-Mongoose.

היא לוקחת את פרטי ההתחברות מתוך משתני סביבה:

- `MONGO_DB`
- `APP_DB_USERNAME`
- `APP_DB_PASSWORD`

היא מתחברת ל-MongoDB, ורק אחרי שהחיבור מצליח היא מתחילה להאזין לבקשות.

### למה זה טוב

כך אני מוודא שהקונטיינר של האפליקציה לא "יעלה חצי" כשהמסד לא זמין.

---

## 7. Health Checks

הוספתי endpoint של `/health` באפליקציה.

ה-endpoint בודק אם החיבור ל-MongoDB באמת פעיל.

אם החיבור תקין:

- מחזיר `200`
- מחזיר סטטוס `ok`

אם החיבור לא תקין:

- מחזיר `503`
- מחזיר סטטוס `degraded`

### למה זה חשוב

Health check טוב עוזר גם ל-Docker Compose וגם ל-CI/CD להבין אם המערכת באמת מוכנה, ולא רק אם התהליך "רץ".

---

## 8. Infrastructure as Code

הגדרתי את כל המערכת כקוד:

- `Dockerfile` לכל רכיב
- `docker-compose.yml` לכל המערכת
- קבצי קונפיגורציה ל-Nginx
- סקריפטי אתחול ל-MongoDB
- workflow של GitHub Actions

כלומר, לא צריך להריץ התקנות ידניות או להגדיר דברים מחוץ לריפו כדי להרים את המערכת.

---

## 9. CI/CD

בנוסף ל-CI, הוספתי גם CD.

### CI

בשלב ה-CI אני:

- יוצר קובץ `.env`
- מריץ `docker compose up --build -d`
- מחכה שהמערכת תהיה healthy
- בודק שהעמוד הראשי מחזיר את הנתון של apples

### CD

בשלב ה-Deploy אני:

- מתחבר ל-EC2 דרך SSH
- מושך את הקומיט האחרון
- יוצר `.env` מתוך GitHub Secrets
- מעלה קודם את MongoDB
- מעדכן או יוצר את משתמש האפליקציה
- מעלה את `app` ואת `nginx`
- בודק שוב health ואת עמוד הבית

### למה זה חשוב

זה מראה שלא רק בניתי אפליקציה, אלא גם תהליך מסודר לבדיקה ופריסה שלה.

---

## 10. סיסמאות והרשאות

אחת ההרחבות שהוספתי היא הפרדה בין:

- Mongo admin credentials
- application DB credentials

המשמעות:

- האפליקציה לא משתמשת ב-root/admin
- הסיסמאות לא נשמרות בקוד
- ב-CD אני מזריק אותן דרך GitHub Secrets

---

## 11. Password Rotation

הוספתי תמיכה בסיסית ברוטציה של סיסמת משתמש האפליקציה.

איך זה עובד:

1. מעדכנים secret ב-GitHub
2. תהליך ה-deploy רץ
3. MongoDB עולה
4. סקריפט `ensure-app-user.js` מעדכן את סיסמת המשתמש של האפליקציה
5. האפליקציה עולה עם הסיסמה החדשה

### למה זה טוב

זה מראה חשיבה תפעולית:

- לא לשמור סודות בקוד
- לא להשתמש במשתמש אדמין באפליקציה
- לאפשר החלפת סיסמה בצורה מבוקרת

---

## 12. גיבוי ושחזור

הוספתי גם יכולת backup/restore ל-MongoDB.

### Backup

יש סקריפט שיוצר גיבוי של המסד לקובץ מכווץ תחת:

`backups/`

### Restore

יש סקריפט נוסף שמשחזר את הדאטה מתוך קובץ גיבוי.

### למה הוספתי את זה

רציתי להראות שלא חשבתי רק על "איך להרים את המערכת", אלא גם על "איך להתאושש ממנה".

זה חשוב במיוחד כי MongoDB הוא רכיב stateful.

## 17. תשובות קצרות לשאלות נפוצות

### למה 3 קונטיינרים?

כדי להפריד תחומי אחריות:

- Nginx כ-entrypoint
- Node.js ללוגיקה
- MongoDB לנתונים

### למה Nginx?

כדי לעבוד עם reverse proxy, לקבל מבנה יותר production-like, ולא לחשוף ישירות את אפליקציית Node.

### למה האפליקציה לא משתמשת במשתמש אדמין?

כי זה פחות בטוח. עדיף משתמש אפליקטיבי עם הרשאות מינימליות.

### איך ה-DB מאותחל?

Mongo מריץ init script בזמן האתחול הראשוני, שמכניס את הנתונים ויוצר את משתמש האפליקציה.

### איך בדקת שהמערכת עובדת?

באמצעות:

- `docker compose up --build`
- `curl /health`
- בדיקת עמוד הבית
- workflow של GitHub Actions

### מה מראה את החשיבה ה-DevOps בפרויקט?

- תהליך CI/CD
- שימוש ב-Secrets
- Health checks
- רוטציית סיסמה
- גיבוי ושחזור
